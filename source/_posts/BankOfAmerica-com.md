I authored and integrated the "Account Transaction Module" UI in 2010-2012, still running in production today, in Bank of America's Checking/Savings ("Deposits") pages, and likely also still on Corporate Credit. The integration is comprised of several artifactually-maintained module-components I also later supported, the bulk of which is the transactions module mentioned. The regexes doing the frontend search validation in JS I wrote then, are usually still running today in the transactions search panel of the Online Banking production browser-experience that millions of customers continue to rely on today, 24x7. The jQuery DataTables integration tying it all together--with its Sort and Filter controls running in the thead of the table--are (like I said, usually, when rookie Java devs don't mess up the regular production app redeployments, occasionally mistakenly thinking these sorts should be server-side, when the algos were authored to run more-virtuosically, instantly, from clicks in the browser UI) are still there, such that complex transactions Sorts do not require browser-reloads from the server-side J2E app.

If you see "PIPAD" acronym-named JS bundles running in the View Source of the page, that's the Java j2e application runtime probably still deployed for the Deposits-related UIs, which came to own my UI code after a corporate re-org, which left several Java applications-teams owning the prior UI Teams' code, scattered across several Java applications. These UI components were standalone at first, and centrally-managed by the UI Team I worked for in 2010-2012, thus were well-documented and centrally-maintained as artifacts in the UI team's source repositories. Then, they were rewritten to accommodate tighter-knit J2EE application-ownership following the corporate re-org, which ultimately'd allowed me to go on a sabbatical, after which I was rehired by the app that owned my code.
